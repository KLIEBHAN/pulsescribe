Performance work log - 2026-03-28

Scope
- Focused on frontend refresh paths outside the last two commits (`bc51fa9`, `c57906e`).
- Left recent overlay, onboarding and hotkey changes untouched unless compatibility required otherwise.
- Kept existing log and transcript behavior functionally unchanged.

Hotspots reviewed
- `ui/settings_windows.py`: transcript auto-refresh re-read `history.jsonl`, reformatted the last 50 entries and replaced the full `QPlainTextEdit` document on every history change.
- `ui/welcome.py`: the macOS transcripts panel did the same work and re-laid out the full `NSTextView` on each refresh.
- `utils/history.py`: transcript history already had efficient tail reads, but no append-aware delta reader or bounded merge helpers for transcript views.

Changes in this pass
- `utils/history.py`
  Added append-only history delta reads, bounded visible-window merging and shared per-entry formatters for Windows and macOS transcript views.
- `ui/settings_windows.py`
  Cached the visible 50 transcript entries, skipped full `history.jsonl` reloads on append-only updates and appended new transcript blocks directly into the viewer when the visible window could stay incremental. When the 50-entry window needs trimming, it now rebuilds from the in-memory cache instead of rereading the file.
- `ui/welcome.py`
  Applied the same cached transcript window strategy, moved safe append-only updates onto `NSTextStorage` and kept the old full-refresh behavior as a fallback for unsafe states.
- Tests
  Added regression coverage for the new history delta helpers and for both transcript incremental refresh paths.

Why this helps
- Removes repeated tail reads, parsing and formatting work from the hot transcript refresh loop in the common append-only case.
- Avoids expensive full document resets on both Qt and AppKit transcript widgets when only new entries were added.
- Preserves the visible 50-entry window and safe fallback behavior for non-append-only or oversized updates.
