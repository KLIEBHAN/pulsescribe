UX notes - 2026-04-21

Latest pass - vocabulary editor UX (`utils/vocabulary.py`, `ui/vocabulary_feedback.py`, `ui/settings_windows.py`, `ui/welcome.py`)
- Focused on the custom vocabulary editors after reviewing their remaining guidance / validation / save-feedback gaps.
- Main weaknesses found:
  1. Limits were mentioned, but not explained in a way that helped users decide what to do.
  2. Duplicate handling and normalization were too opaque, especially across comma-separated input and case variants.
  3. Save vs. unchanged states were weak or inconsistent.
  4. Empty states and input guidance did not clearly explain what belongs in the vocabulary editor.
  5. Windows and macOS vocabulary surfaces behaved and explained themselves differently.
- Changes in this pass:
  - added shared vocabulary parsing + analysis helpers so editor feedback can use the same normalization rules as persistence
  - added shared vocabulary UX copy helpers for empty states, live edit feedback, load feedback and save / unchanged states
  - improved Windows vocabulary editor copy, placeholder text, tooltips, accessibility names and button labels
  - added live Windows editor feedback for duplicates, pending clears, limits and unchanged states
  - normalized the Windows editor text after save so users immediately see the persisted result
  - improved macOS Welcome vocabulary guidance and bottom status copy, including duplicate / empty / unchanged feedback
  - wired live vocabulary warning updates on macOS while editing
- Validation:
