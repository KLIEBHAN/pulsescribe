UX notes - 2026-04-19

Current pass focus
- Intentionally avoided the areas touched in the last two commits:
  - `ui/settings_windows.py`
  - prompt-related parts of `ui/welcome.py`
  - `utils/custom_prompts.py`
- Worked instead on onboarding / first-run guidance and shared setup UI:
  - `ui/onboarding_wizard_windows.py`
  - `ui/permissions_card.py`
  - `ui/hotkey_card.py`
  - `ui/styles_windows.py`

Main UX weaknesses reviewed first
1. Windows onboarding had several unclear interaction patterns:
   - Fast mode showed technical API-key handling with weak guidance.
   - language selection used raw codes instead of readable names.
   - hotkey capture relied on clicking read-only fields, which was easy to miss and poor for keyboard accessibility.
   - test dictation copy suggested hotkey-driven behavior although the Windows flow is primarily button/IPC-driven.
   - summary values were too raw (`deepgram`, `auto`, lowercase hotkeys) and did not clearly confirm whether the dictation test succeeded.
2. Shared macOS setup cards still communicated optional permissions too aggressively:
   - Accessibility/Input Monitoring looked like hard errors instead of optional/recommended capabilities.
   - action labels like `Open` were vague.
3. Windows visual feedback was inconsistent:
   - weak focus visibility on buttons/choice cards
   - sparse empty states in onboarding cards
   - minimal recovery guidance after test / permission / hotkey issues

What changed in this pass
1. Windows onboarding (`ui/onboarding_wizard_windows.py`)
- Goal selection
  - clearer copy for Fast / Private / Advanced
  - readable language names instead of raw language codes
  - extra hint explaining when `Automatisch` is preferable
  - stronger Fast-mode API-key guidance and automatic focus when input is required
- Permissions step
  - clearer expectation-setting for Windows microphone access
  - better status wording and feedback when Windows settings open/fail to open
  - stronger troubleshooting hint for failed recordings
- Hotkey step
  - added explicit `Aufnehmen` / `Entfernen` buttons for toggle + hold rows
  - preserved click-on-field behavior, but made the main action much clearer
  - improved per-row explanations for Toggle vs Hold
  - added clearer recording microcopy (`Enter` / `Speichern` / `Esc`)
  - improved button state changes during active hotkey capture
  - improved success / preset / removal feedback text
- Test dictation step
  - clarified that the Windows test is a safe in-wizard recording and does not paste anything
  - improved initial empty state, button labels and troubleshooting copy
  - improved success / no-speech / error messages
  - hotkey reminder is now formatted and easier to scan
- Summary step
  - friendlier provider / language / hotkey formatting
  - added explicit dictation-test status (`Erfolgreich geprüft`, `Noch nicht getestet`, etc.)

2. Shared permissions card (`ui/permissions_card.py`)
- simplified description text
- changed vague `Open` actions to `Settings`
- distinguished required vs optional capabilities more clearly:
  - microphone = required
  - accessibility = recommended for auto-paste
  - input monitoring = needed for Hold / some hotkeys
- reduced alarmist wording for optional missing permissions

3. Shared hotkey card (`ui/hotkey_card.py`)
- clearer empty-state placeholder (`Use Record…`)
- better hint text for Toggle vs Hold behavior
- added a neutral default status when no hotkeys are configured yet

4. Shared Windows styling (`ui/styles_windows.py`)
- better focus visibility for buttons and choice cards
- placeholder text styling for inputs/editors
- stronger keyboard-navigation affordances without changing behavior

Why this improves UX
- onboarding is easier to understand on first use
- the next action is clearer on every step
- hotkey capture is more discoverable and more keyboard-friendly
- loading / empty / failure states provide better recovery guidance
- summary information now confirms setup outcomes instead of exposing raw config values
- optional permissions no longer feel like blocking errors
