Performance work log - 2026-03-28

Scope
- Left the last two commit areas untouched: transcript/log incremental refresh and permission/overlay repaint work.
- Focused on `.env` persistence churn in Windows settings/local preset flows and setup overview UI churn.
- Preserved current onboarding, settings, preset, and API key behavior.

Current weaknesses reviewed
- `ui/settings_windows.py`: `Save && Apply` rewrote `.env` once per setting and once per API key, causing a long chain of small disk reads/writes in one UI action.
- `utils/presets.py`: applying a local preset performed many separate `.env` mutations even though the user triggered one atomic configuration change.
- `ui/settings_windows.py`: setup overview labels/styles and mode-dependent card visibility were reapplied even when the rendered result had not changed, causing avoidable repaints and layout work.

Changes in this pass
- `utils/preferences.py`
  Added `update_env_settings()` to batch multiple `.env` mutations into one atomic read/write pass, preserve unrelated lines, collapse duplicate updated keys, and skip unchanged rewrites.
- `ui/settings_windows.py`
  Switched the Windows settings save path to one batched `.env` update for settings and API keys, and deduped repeated setup overview/status/visibility widget updates.
- `utils/presets.py`
  Switched local preset application to the batched `.env` update path.
- Tests
  Added regression coverage for batched env writes, no-op env updates, preset batching, canonical FP16 handling in batched saves, and batched API key persistence.

Why this helps
- `Save && Apply` no longer blocks on a long series of tiny `.env` writes, so the settings window responds faster and feels less sticky.
- Local preset application now lands as one atomic config update instead of many serial writes.
- Setup overview refreshes and mode switches avoid redundant text/style/visibility churn, which trims repaint/layout pressure in a frequently hit UI path.
