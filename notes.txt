Guardrails
- Focused on UI-heavy paths outside the last two commits.

- Kept onboarding flows, hotkey handling and overlay behavior functionally unchanged.

Previous pass already completed
- `ui/welcome.py`: lazy tab building for heavy macOS settings tabs.
- `ui/welcome.py`: prevented Logs auto-refresh from starting before the Logs UI exists.
- `ui/overlay_windows.py`: skipped duplicate label and bar-color updates in the Windows Tk overlay.

Hotspots reviewed in this pass
- `ui/onboarding_wizard.py`: macOS onboarding eagerly built every step at window creation, including permissions, hotkey, transcript and summary UI.
- `ui/onboarding_wizard_windows.py`: Windows onboarding built every `QStackedWidget` page up front even though only one page is visible.
- `ui/overlay_pyside6.py`: active recording/interim updates recreated fonts and reapplied identical label styles/text in a high-frequency path.

Current weaknesses identified
- Unnecessary first-open work in both onboarding flows from constructing hidden step trees before the user reaches them.
- Hidden-control mutations in the macOS onboarding hotkey path depended on eager build and would become fragile once step construction was deferred.
- Repeated `QLabel.setFont(...)`, `QLabel.setStyleSheet(...)` and `QLabel.setText(...)` calls in the PySide overlay added avoidable churn during interim updates.

Changes in this pass
- Added lazy step building to `ui/onboarding_wizard.py` so only the active macOS onboarding step is created initially; other steps materialize on first navigation.
- Added lazy page building to `ui/onboarding_wizard_windows.py` so the `QStackedWidget` only receives a page when that step is first shown.
- Kept macOS hotkey actions safe by forcing the Hotkey step to build before presets or recording actions mutate hidden controls.
- Cached PySide6 overlay label font/style/text state so duplicate interim updates skip redundant widget mutations.
- Added regression tests for lazy onboarding step/page creation and the new PySide overlay no-op guards.

Why this helps
- Faster onboarding open time on both platforms because hidden cards, transcript widgets and summary controls no longer build eagerly.
- Lower initial UI memory/setup cost when resuming the wizard on an early step.
- Lower steady-state overlay overhead on Windows/PySide during recording because label styling work is only done when something actually changes.

Validation
- `pytest tests/test_onboarding_wizard_flow.py tests/test_onboarding.py tests/test_onboarding_windows.py tests/test_overlay.py tests/test_overlay_pyside6.py -q`

Next worthwhile areas

- `ui/welcome.py`: the Logs/Transcripts area still does its data load when the tab is first built; segment-level lazy loading would trim that path further.
- `ui/onboarding_wizard.py` and `ui/onboarding_wizard_windows.py`: if profiling still shows first-open cost, the heavy steps themselves can be split into smaller lazy sections.
