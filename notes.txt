Performance work log - 2026-03-28

Scope
- Focused on frontend-adjacent startup and settings performance.
- Avoided the areas changed in the last two commits (`audio/recording.py`, provider/env diagnostics code).

Hotspots identified
- Repeated `preferences.json` reads on startup/UI paths caused avoidable disk I/O.
- `SettingsWindow._load_settings()` re-read environment-backed settings key-by-key instead of working from one snapshot.
- Settings bulk initialization and local preset application triggered redundant setup-overview recomputations.
- The Prompts tab loaded prompt data eagerly even when the tab was never opened.

Changes made
- Added file-signature caching for `preferences.json` in `utils/preferences.py`.
- Reworked Windows settings loading to use one `read_env_file()` snapshot plus cached process-env API-key fallbacks.
- Batched setup-overview refreshes during bulk UI updates so the expensive status recomputation only runs once per load/apply cycle.
- Deferred prompt loading until the Prompts tab is actually opened.
- Added regression tests for the new caching and deferred-load behavior.

Why this helps
- Faster startup/settings open paths due to less filesystem churn.
- Less unnecessary UI work while the settings form is populated.
- Better responsiveness when applying local presets.
- Reduced eager work for tabs the user may never open.

Validation
- `pytest tests/test_preferences_cache.py tests/test_settings_windows_load.py tests/test_settings_windows_prompts.py -q`
- `pytest tests/test_settings_windows_logs.py tests/test_settings_ui.py -q`

Next worthwhile areas
- macOS `WelcomeController` still does substantial eager tab construction and would benefit from the same lazy-loading strategy.
- Overlay rendering can be profiled next for paint-path allocations and label/style churn under long recording sessions.
- Large AppKit/PySide tab builders could be lazily created on first activation if window-open latency still feels high.
