Performance work log - 2026-03-28

Scope
- Focused on frontend-heavy refresh paths outside the last two commits.
- Left the most recent `overlay`, `onboarding` and `hotkey` changes untouched for this pass unless compatibility required otherwise.
- Kept existing log, transcript, onboarding and settings behavior functionally unchanged.

Hotspots reviewed
- `ui/welcome.py`: the macOS logs view rebuilt the full visible tail buffer whenever the log file grew.
- `ui/settings_windows.py`: the Windows logs view replaced the whole `QPlainTextEdit` document on each log change, even for small append-only updates.
- Both paths sit behind auto-refresh timers, so repeated full document swaps added unnecessary render work during active sessions.

Changes in this pass
- `utils/log_tail.py`
  Added shared helpers to read only the appended file region and to merge append-only updates back into fixed line/character windows.
- `ui/settings_windows.py`
  Enabled a bounded log document block count and switched append-only log growth at the bottom of the view to incremental cursor inserts instead of full `setPlainText()` rebuilds.
- `ui/welcome.py`
  Switched append-only log growth at the bottom of the macOS logs view to incremental text storage appends, with fallback to the existing full refresh path when truncation, rotation or scroll state makes incremental updates unsafe.
- Tests
  Added regression coverage for the new tail helpers and for both incremental log refresh paths.

Why this helps
- Reduces expensive full-text widget rebuilds on the hottest auto-refreshing log paths.
- Cuts repeated text layout and repaint work when logs are simply growing at the end, which is the common case during active sessions.
- Keeps the previous safe full-refresh behavior for rotated, truncated or large-jump updates.

Validation
- `pytest tests/test_settings_windows_load.py tests/test_settings_windows_logs.py tests/test_settings_ui.py tests/test_log_tail.py -q`
- `python -m py_compile ui/settings_windows.py ui/welcome.py utils/log_tail.py tests/test_settings_windows_logs.py tests/test_settings_ui.py tests/test_log_tail.py`

Next worthwhile areas
- `ui/settings_windows.py`: transcripts still rebuild the full formatted history when `history.jsonl` changes; append-aware history updates are the next clear win.
- `ui/welcome.py`: transcripts still rebuild the visible history payload on each change; incremental entry formatting would reduce redraws there too.
- `ui/welcome.py`: once the visible log tail re-truncates past the fixed 15k window, the code still falls back to a full refresh; if profiling still points there, bounded incremental re-trimming would be the next refinement.
