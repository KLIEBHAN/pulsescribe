Performance work log - 2026-03-28

Scope
- Kept the transcript/log refresh work from `49efeb1` and `bc51fa9` untouched.

- Preserved existing onboarding, settings and overlay behavior.

Current weaknesses reviewed
- `ui/permissions_card.py`: every refresh rebuilt status colors and used the color objects in the cache key. On real AppKit objects that can defeat the dedupe path and reapply the same text/color updates repeatedly.
- `ui/permissions_card.py`: `after_refresh` ran on every auto-refresh tick, which could retrigger expensive parent renders even when no permission state changed.
- `ui/overlay_windows.py`: Tk canvas bars updated all coordinates every frame, including subpixel changes with no visible payoff.
- `ui/overlay_pyside6.py`: the animation loop requested a full repaint on every tick even when smoothed bar heights had barely moved.

Changes in this pass
- `ui/permissions_card.py`
  Reused a lazy status-color palette, cached status updates by stable semantic keys instead of transient color objects, and only ran `after_refresh` when the card actually changed.
- `ui/overlay_windows.py`
  Added per-bar drawn-height tracking and skipped canvas coordinate updates for subpixel-only height deltas.
- `ui/overlay_pyside6.py`
  Added painted-height tracking and only requested a repaint once bar movement crossed a visible threshold or the overlay state changed.
- Tests
  Added regression coverage for stable permission-card dedupe with fresh color objects, change-only `after_refresh` behavior, and the new overlay subpixel skip paths.

Why this helps
- Cuts redundant AppKit widget mutations and avoids unnecessary onboarding re-renders during permission polling.
- Reduces Tk canvas work in the hottest Windows overlay loop.
- Lowers avoidable Qt repaint pressure without changing the visible animation behavior.
