Latest pass - overlay animation hot paths

  - `ui/overlay.py`
  - `ui/overlay_windows.py`
  - `ui/overlay_pyside6.py`

- Main hotspots identified:
  1. Windows/PySide recording animations still asked `AnimationLogic` for one bar at a time every frame, which repeated Python method/call overhead on the hottest 60 FPS path.
  2. macOS overlay animation frames still re-read fixed wave-view geometry from AppKit and rebuilt non-recording frame vectors bar-by-bar on the main thread.
  3. Windows Tk overlay still recomputed static bar layout data during frame rendering even though those positions never change after setup.
- Changes in this pass:
  - `ui/animation.py`
    - added frame-level APIs for normalized values/heights so callers can consume one cached vector per frame instead of repeatedly calling per-bar helpers.
  - `ui/overlay_windows.py`
    - switched the render loop to the batch height API and cached static bar x-positions/center once.
  - `ui/overlay_pyside6.py`
    - switched the animation loop to the batch height API so recording frames no longer repeat per-bar cache/key work.
  - `ui/overlay.py`
    - reused cached wave geometry on hot frame paths and routed listening/transcribing/done frame building through one batch helper.
- Why this helps:
  - lowers Python call overhead on 60 FPS recording loops
  - removes redundant per-frame geometry/layout recomputation
  - trims main-thread work on the macOS overlay and CPU work on the Windows overlays without changing visuals

Current frontend weaknesses reviewed
1. Daemon result polling / overlay feedback
- `pulsescribe_daemon.py` still drained result-queue bursts one message at a time.
- Audio level traffic is much hotter than status/result traffic, but every queued audio-level message still triggered its own overlay update during a poll tick.
- Long or bursty recording periods could therefore spend unnecessary main-thread work on redundant overlay audio updates.
- The previous safety break limited each tick to 50 messages, which also made backlog clearing slower once many audio messages accumulated.

2. Permission state reads during onboarding / setup bursts
- The app already reuses the live `PermissionsCard` snapshot when available, but fallback reads still call three separate macOS permission helpers.
- Those helpers are comparatively expensive and can be hit repeatedly during short onboarding/setup render bursts.
- There was no short-lived shared snapshot for these immediate repeat reads.

Changes in this pass
1. `pulsescribe_daemon.py`
- Extracted result-queue batch processing into a dedicated helper.
- Coalesced bursty `AUDIO_LEVEL` messages so one poll tick now forwards only the newest visible level to the overlay.
- Preserved VAD responsiveness by remembering whether any coalesced level crossed the trigger threshold, even if the final level in the burst dropped again.
- Flushed pending audio before later status transitions in the same tick, so ordering stays sane when a burst is followed by `TRANSCRIBING`/other status updates.
- Increased the per-tick drain budget now that hot audio traffic is coalesced, which reduces backlog pressure without spamming UI updates.

2. `utils/permissions.py`
- Added a tiny shared permission-signature cache for immediate repeat reads.
- Added explicit cache invalidation on permission-request/open-settings paths so user-triggered permission changes do not stay stale.

3. `ui/onboarding_wizard.py`
- Routed the non-card fallback permission read through the new shared permission-signature helper.

Tests added/updated
- `tests/test_local_feedback.py`
  - burst audio levels collapse to one overlay update
  - pending audio is flushed before later status transitions in the same poll tick
- `tests/test_permissions.py`
  - shared permission snapshot is reused only within the short cache window
  - permission-request flow invalidates the shared snapshot cache

Why this helps
- Recording responsiveness improves because the hottest queue traffic now scales with the latest visible state instead of every intermediate audio sample delivered to the daemon.
- Overlay/UI work on the main thread is lower during active dictation bursts.
- Permission-heavy onboarding/setup bursts avoid some duplicate macOS permission checks while still refreshing quickly after real user-triggered permission changes.
