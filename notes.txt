UX notes - 2026-04-21

Latest pass - active dictation overlay feedback clarity (`ui/overlay_feedback.py`)
- Focused on the everyday active-dictation overlay flow next, instead of revisiting the recent transcript-history or onboarding work.
- Main weaknesses found:
  1. The overlay still showed terse or technical raw status/error text in a very small single-line surface, which made loading and failure states harder to parse quickly.
  2. The recording fallback label was too generic, so brief no-interim moments looked less intentional than they should.
  3. Optional RTF/speed feedback used a cryptic `✓ (0.3x)` shorthand that was hard to understand without prior context.
  4. Because PySide accessibility descriptions mirror overlay text, unclear overlay copy also reduced accessibility quality.
- Changes in this pass:
  - added shared normalization for overlay loading/error feedback so common technical details are turned into concise user-facing copy
  - made loading/warmup text calmer and more consistent for narrow overlay layouts
  - replaced the cryptic RTF checkmark shorthand with readable success copy like `Transcript pasted • 0.3x realtime`
  - improved the fallback recording label so short gaps before interim text still feel guided and intentional
  - kept the existing overlay state machine and animations intact while making the visible copy clearer across macOS AppKit, Windows Tk and PySide overlays

Latest pass - transcript history panel clarity across Welcome + Windows Settings (`ui/welcome.py`, `ui/settings_windows.py`, `ui/logs_panel_feedback.py`)
- Focused on an everyday transcript/settings flow outside onboarding, instead of revisiting the two recent onboarding passes.
- Main weaknesses found:
  1. The Windows transcript panel overloaded one label for both entry count and action/error feedback, so context disappeared after refresh or clear actions.
  2. Transcript history stayed visually under-explained: it was not obvious enough that entries are local-only and what Clear History actually affects.
  3. Destructive `Clear History…` actions stayed available even when there was nothing useful to clear, which added noise and hesitation.
  4. Transcript panel hierarchy differed between macOS Welcome and Windows Settings more than necessary.
- Changes in this pass:
  - added shared transcript-panel hint copy that explains the history is stored locally and what clearing does
  - separated persistent transcript count from transient action/error feedback in the Windows Settings transcript panel
  - disabled transcript-history clear actions when no local entries are available, while still allowing recovery from load-error states
  - added a small persistent transcript-history hint line in the macOS Welcome history panel for better guidance and consistency
  - kept existing refresh / clear behavior intact while making the visible state and destructive affordances clearer

Latest pass - macOS onboarding practice-dictation flow (`ui/onboarding_wizard.py`, `pulsescribe_daemon.py`)
- Focused on the central macOS onboarding test step next, instead of reopening the recent Windows onboarding or settings feedback work.
- Main weaknesses found:
  1. The practice step was effectively hotkey-only even though start/stop callbacks already existed, which hurt discoverability and accessibility.
  2. Recording / processing phases reused sparse status text and a mostly blank transcript area, so the flow felt uncertain while nothing was pasted yet.
  3. Cancelled, skipped, empty and error outcomes were not clearly separated in either inline feedback or the final summary.
  4. Button / CTA guidance was too implicit for a first-run onboarding moment on macOS.
- Changes in this pass:
  - added explicit in-step `Start practice` / `Stop` controls on macOS and wired them to the safe onboarding dictation callbacks
  - upgraded daemon callback wiring so button-driven practice runs now receive the same recording / processing state feedback as hotkey-driven runs
  - added richer test-state handling for starting, recording, processing, cancelled, skipped, empty, success and error outcomes
  - kept the transcript area informative during all transient states instead of leaving it mostly blank
  - added a dedicated helper notice line with clearer next-step guidance and more actionable error feedback
  - improved CTA states so users now see `Start practice`, `Try again`, `Run again`, `Cancel`, `Stop` and `Working…` at the right moments
  - improved final summary wording so cancelled / skipped outcomes read as neutral states instead of generic warnings

Latest pass - Windows onboarding test-dictation guidance + retry flow (`ui/onboarding_wizard_windows.py`)
- Focused deliberately on the onboarding practice/test step instead of reopening the recent settings feedback work.
- Main weaknesses found:
  1. The test step opened with a failure-oriented warning instead of calm step-by-step guidance.
  2. While waiting for the daemon ack, the UI hid the start action and showed a disabled stop button, which left users stuck on slow starts.
  3. Cancelled, no-speech and error outcomes reused an almost empty transcript area, so the result felt broken rather than explained.
  4. Retry, cancel and summary wording collapsed too many different outcomes into generic states.
- Changes in this pass:
  - added explicit state copy for idle, connecting, recording, processing, cancelled, no-speech, success and error test states
  - kept the transcript area informative with clear empty / processing / cancelled / no-speech text instead of blank space
  - turned the pre-recording stop action into an explicit cancel path so slow or unavailable daemon handoff is less confusing
  - added adaptive CTA labels like `Test starten`, `Erneut testen` and `Nochmal testen`
  - kept the contextual helper notice visible and state-aware during critical handoff moments instead of hiding it
  - improved summary feedback so skipped / cancelled / attention-needed states are distinguishable from a successful test
  - normalized common test errors into more actionable user-facing wording
